### What We Accomplished ✅
- **Diagnosed release workflow race condition**
  - MCP server and ESLint plugin releases triggered simultaneously
  - MCP pushed first, ESLint push failed with "non-fast-forward"
  - npm publish succeeded but git state was out of sync
- **Manually synced ESLint plugin release**
  - Used `release-it 1.1.0 --no-npm --ci` to generate changelog/commit/tag
  - Created GitHub Release via `gh release create`
  - Package now properly shows in releases alongside mcp-server
- **Implemented concurrency controls**
  - Added shared `npm-releases` concurrency group to all 3 workflows
  - Added `workflow_run` trigger for mcp-server to run after src completes
  - Removed redundant src path triggers from mcp-server (now handled by workflow_run)
- **Updated NPM_PUBLISHING.md documentation**
  - Added "Workflow Ordering & Concurrency" section
  - Documented release priority order (src → mcp → eslint)
  - Added pre-release behavior notes
  - Added common issue #5 for git push rejection recovery
- **Created issue and PR**
  - Issue #505: https://github.com/movableink/ui/issues/505
  - Draft PR #506: https://github.com/movableink/ui/pull/506

### Challenges Encountered 🔧
- Understanding the race condition required reading GitHub Actions logs to see the "non-fast-forward" rejection
- ESLint plugin was in partial state (npm published, git not synced) - required manual recovery with release-it

### Most Valuable Collaboration
User's question about pre-releases led to documenting the behavior clearly - prereleases share the same concurrency group but don't cascade via workflow_run, so they can be triggered independently per package.

### Key Insight
GitHub Actions concurrency groups solve the race condition, but `workflow_run` triggers provide proper dependency ordering. The combination ensures mcp-server always has fresh src data without racing.
