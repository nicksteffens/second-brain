### What We Accomplished
- Reviewed existing git-ai research doc and added self-hosted deployment concerns (OAuth analysis, data residency, revocability)
- Added pre-enablement testing plan with 5 phases and a test results log
- Captured Phase 1 baseline (clean machine state before install)
- Installed git-ai 1.1.19, immediately audited what it changed
- Caught default config shipping with no `allow_repositories` — was tracking all repos including personal
- Caught org name mismatch (`movable-ink` vs `movableink`) in allowlist patterns
- Locked config to `movableink/*` only, trashed `.git/ai/` from personal repo
- Ran Phase 2 test in UI repo — confirmed tracking works (notes written, agent/model attribution correct, prompts not uploaded)
- Reverted test commit in UI repo
- Audited Claude Code settings.json vs example — no migration needed

### Challenges Encountered
- Default git-ai config was wide open — no `allow_repositories` set on install, meaning personal repos were trackable from moment of install until manually configured

### Most Valuable Collaboration
Security audit of the git-ai install. Catching the default config issues (no allowlist, wrong org name) before data leaked to the server was the key win. The phased testing approach gave us structured verification rather than hoping it works.

### Key Insight
Always audit after install. Default configs aren't safe — git-ai tracked everything out of the box. The install script doesn't ask about repo scoping, doesn't warn about global tracking, and sets `api_base_url` to their public cloud rather than the self-hosted instance you OAuth'd into.
