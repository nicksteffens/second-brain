### What We Accomplished
- Diagnosed why canary releases published as `2.43.1-canary.67` instead of `3.17.0-canary.X`
- Root cause: lerna v9's `git describe --first-parent` skips tags on merged branches, resolving to old `v2.43.0` instead of `v3.16.0`
- Created bug card [sc-193906](https://app.shortcut.com/movableink-epd/story/193906) under Frontend 2026 - Infra epic
- Fixed release workflow with `--include-merged-tags` (correct tag resolution) and `--force-publish` (all packages in canary)
- Added configurable `canary_dist_tag` workflow input (defaults to `canary`, overridable per trigger)
- Updated README to reflect the fix
- Verified fix end-to-end: triggered canary with custom `nick` dist-tag, published as `3.16.1-nick.4` (correct 3.x range)
- Opened PR [#1126](https://github.com/movableink/studio-framework/pull/1126) with verification comment

### Challenges Encountered
- Local lerna v4 vs CI lerna v9 muddied the initial investigation — `lerna --version` locally showed v4 despite lockfile resolving v9

### Most Valuable Collaboration
Root cause analysis — systematically narrowing from "version looks wrong" to the exact `git describe --first-parent` behavior difference, then finding the existing `--include-merged-tags` flag that solves it.

### Key Insight
`git describe --first-parent` is invisible to most developers but critically affects lerna's canary versioning. Tags on merge commits aren't reachable via first-parent traversal, which breaks version resolution on feature branches in repos with merge-based release workflows.
