### What We Accomplished
- Investigated Shortcut story sc-192830 — mise as Volta replacement for Node version management
- Tracked down the Docker image source: `movableink/docker-node-with-chrome` repo, `add-volta-image` branch (never merged to master)
- Read the existing Volta Dockerfile — `debian:stable-slim` + Chrome + `curl https://get.volta.sh | bash`
- Discovered the repo README claims images moved to `movableink/provisioning` but the `node-with-chrome` directory doesn't exist there
- Analyzed all 7 CI workflows in front-end-infra that depend on the `:volta` Docker image
- **Key finding:** Volta is dead weight in CI — every workflow uses `actions/setup-node@v4` with hardcoded `node-version: '20'`, overriding Volta entirely. The `$HOME/.volta` cache path is unused.
- Identified existing version drift: `package.json` pins `volta.node: 20.19.3` but CI hardcodes `node-version: '20'`
- Designed phased migration strategy: new Docker image with mise, `.mise.toml` as single source of truth, `jdx/mise-action` replacing `setup-node`, parallel rollout starting with `ember.lint.yml`
- Created comprehensive Notion page in Front-End Infrastructure wiki: "Mise as Volta Replacement: CI Docker Image Migration"
- Page includes current state analysis, proposed Dockerfile, workflow changes, rollout plan, comparison table, and risks

### Challenges Encountered
- Volta Dockerfile lives on an unmerged branch (`add-volta-image`) of a deprecated repo — required branch-specific API queries to find and read it

### Most Valuable Collaboration
Migration strategy design — the phased rollout plan and architectural decisions around mise + corepack for Yarn, with `.mise.toml` as single source of truth for local and CI

### Key Insight
Volta isn't actually managing Node in CI — `setup-node` overrides it in every workflow. The Docker image is really just "Chrome + Debian base packages." This means the migration is lower risk than expected since Volta isn't load-bearing in CI.
