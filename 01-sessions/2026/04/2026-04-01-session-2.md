### What We Accomplished
- Debugged the full tunnel chain: VPN connectivity, SocketProxy header serialization, IPv6 binding, Vite allowedHosts, and `express-http-proxy` FakeSocket incompatibility
- Discovered `reverse_proxy` approach from RFC doesn't work — SocketProxy can't serialize complex Rack headers into protobuf
- Pivoted to rendering `vite.html.erb` with `@studio_url` set to the tunnel URL — much cleaner, assets load directly from tunnel
- Built `tunnel_proxy` helper in Rails using `Net::HTTP` with minimal headers for future Ember support
- Got Studio loading locally through the tunnel — Vite assets streaming through SocketProxy against local Rails
- Added improved CLI output with local URL, share URL, tunnel ID for easy copy-paste
- Created RFC sub-page documenting all 7 issues discovered and revised architecture
- Pushed updated Rails PR with all fixes

### Challenges Encountered
- VPN required to reach tunnel IPs (10.0.x.x) — not obvious initially
- SocketProxy protobuf serialization fails on complex headers from `rails-reverse-proxy`
- Vite binds to IPv6 (::1) on macOS but CLI was connecting IPv4 (127.0.0.1)
- Vite `allowedHosts` blocks requests with tunnel hostname in Host header
- `express-http-proxy` stream piping incompatible with SocketProxy's FakeSocket
- Staging 504 — unresolved network routing issue

### Most Valuable Collaboration
Persistent debugging — systematically worked through 7+ issues, each revealing the next layer of the problem. Every fix taught us something about the architecture that the RFC couldn't predict.

### Key Insight
Live testing is essential — every assumption from the RFC was wrong in practice. The `reverse_proxy` approach, the `VITE_TUNNEL_URL` env var, IPv4 connectivity — all needed to be discovered through real debugging. The winning `vite.html.erb` approach was found by accident while eliminating what didn't work.
